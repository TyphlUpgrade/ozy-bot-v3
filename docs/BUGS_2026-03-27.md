## Bugs Found

### BUG-1: CRWD 127-Second Exit — ALREADY FIXED (Misdiagnosed Initially)

**Observed:** CRWD swing short entered at 14:45Z ($369.76, stop $385). Exited 127 seconds later at $370.75 (-0.22%).

**Initial misdiagnosis:** Log analysis initially suggested gate expiry bypassed unfulfilled entry conditions.

**Actual root cause (confirmed from raw logs):** Entry conditions were `rsi_accel_max: -0.50` (RSI deceleration gate for shorts). CRWD was deferred 3 cycles while `rsi_accel_3` remained above -0.50. On cycle 4, `rsi_accel_3` cleared the gate — entry was legitimate and correct. The exit at $370.75 was triggered by the old direction-unaware stop check: `price <= stop` → `370.75 <= 385.00 = True`. The exit log shows `"Price 370.75 at/below stop 385.00"` — the `at/below` phrasing is from the old long-direction code path.

**Status: Fixed in today's commit (`1d7bf4d`).** Direction-aware stop check now uses `price >= stop` for shorts. Gate expiry logic is working correctly and was not involved.

**No further action needed.**

### BUG-2: ATO Order Timeout Inconsistency

**Observed:** ATO limit buy placed at 15:17:20Z with strategy=swing. Cancellation fired at 15:37:22Z citing `age=1201s timeout=300s`.

**Expected behavior:** `swing_limit_order_timeout_sec=1200` → cancel at 1200s (20 min). The order was cancelled at 1201s which is correct for the swing timeout, but the log message says `timeout=300s` (the momentum timeout). The order appears to have been cancelled at the *right* time (just over 1200s) but logged with the wrong timeout value.

**Root cause:** Likely the timeout value logged in the cancellation message is reading `limit_order_timeout_sec` (300) rather than `swing_limit_order_timeout_sec` (1200) when formatting the warning string. The actual cancellation behavior is correct.

**Severity:** Low. Operational behavior correct, log message misleading. Fix: use strategy-specific timeout value in cancellation warning log.

### BUG-3: Implicit Rejections — BG and SO Not Evaluated

**Observed:** BG and SO were passed to Claude as existing open positions (correctly), but also appeared in the watchlist tier-1 candidates (both have priority_tier=1). The orchestrator logged "Implicit rejection: BG/SO sent to Claude but absent from both new_opportunities and rejected_opportunities."

**Root cause:** Open position symbols are included in the watchlist candidates section of the context. Claude sees them as positions (with full hold/exit review) and doesn't re-evaluate them as new entry candidates — so they never appear in new_opportunities or rejected_opportunities. The implicit rejection log is a false alarm for open positions.

**Severity:** Low / noise. No trading impact. Fix: exclude symbols already held as positions from the watchlist candidate block before passing to Claude, OR suppress the "implicit rejection" warning for symbols that appear in `portfolio.positions`.
