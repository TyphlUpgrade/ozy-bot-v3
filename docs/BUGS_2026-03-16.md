## Bug 1 — Re-adoption runaway loop

**Severity:** Critical
**Evidence:** 16 AMD `sell` market orders placed in 3 minutes (15:07:56–15:10:42), all filled ~10s apart. AMD ended the session with a 30-share short position the bot did not intend.

**Root cause:** After `_journal_closed_trade` removed AMD from local portfolio, `_fast_step_position_sync` (step 5) ran 10 seconds later, saw AMD still at broker (position settles, and selling a short deepens it), and re-adopted it. The re-adopted position had `direction="long"` (see Bug 3), causing `_fast_step_quant_overrides` to fire a new SELL order. The cycle repeated 16 times.

**Culprit:**
- `orchestrator.py:959` — `_fast_step_position_sync` adoption path had no guard against re-adopting a recently-closed symbol.

**Fix:** Added `_recently_closed: dict[str, float]` tracking close timestamps. Adoption skips any symbol closed within the last 60 seconds. Both `_journal_closed_trade` (line 833) and the ghost cleanup path (line 1043) populate `_recently_closed`.

## Bug 2 — Claude exit recommendations silently ignored

**Severity:** Critical
**Evidence:** `portfolio.json` shows 6 consecutive `review_notes` on the AMD short position from 15:12 to 16:42 ET, each stating the short thesis had failed. No exit order was ever placed.

```
[15:14] "Short thesis failing as AMD consolidates above $198 support"
[15:19] "Short thesis has failed..."
[15:24] "Short thesis completely failed..."
[16:05] "Short thesis completely invalidated..."
[16:42] "Short thesis completely invalidated..."
```

**Root cause:** `_apply_position_reviews` only appended notes and adjusted targets. The `action` field from Claude's response (`"exit"`) was read but never acted upon.

**Culprit:**
- `orchestrator.py:1883` — `_apply_position_reviews` had no branch for `action == "exit"`.

**Fix:** When `action == "exit"`, the function now places a market exit order immediately (buy-to-cover for shorts, sell for longs) and records it in fill_protection.

## Bug 3 — Startup reconciliation always assigned direction="long"

**Severity:** High
**Evidence:** AMD journal entry shows `direction="long"` despite the broker holding a short position. This caused `_fast_step_quant_overrides` to skip the short-position guard (`if position.intention.direction == "short": continue`) and fire repeated sell orders.

**Root cause:** `startup_reconciliation` created adopted positions with `TradeIntention()` (default constructor), which sets `direction="long"` regardless of what the broker reports.

**Culprit:**
- `orchestrator.py:295` — `startup_reconciliation` step 1, adopted position block used bare `TradeIntention()`.

**Fix:** Now infers direction from `broker_pos.side`: `"short"` if side is `"short"` or `"sell"`, else `"long"` (line 339).

## Bug 4 — Short open fills misrouted as trade closes

**Severity:** High
**Evidence:** META trade journal entry #1 (15:12:13–15:12:20, pnl=0%, exit_price=628.23) shows a position opened and immediately closed at the same price. The position was adopted by position_sync before the opening fill was detected; when the SELL fill was then processed, `_dispatch_confirmed_fill` saw a position existed and called `_journal_closed_trade` instead of `_register_opening_fill`.

**Root cause:** Fill dispatch routed by `change.side`: `side=="sell"` → `_journal_closed_trade`. This is wrong for short opens (a sell that opens a new short position).

**Culprit:**
- `orchestrator.py:674` — `_dispatch_confirmed_fill` originally branched on `change.side` instead of portfolio state.

**Fix:** Routes by portfolio presence: position exists → closing fill → journal; no position → opening fill → register. Correct for all four cases (long open, long close, short open, short close).

## Bug 5 — SPY hallucination loop (ghost cleanup race)

**Severity:** High
**Evidence:** Three SPY journal entries in 31 seconds (15:25:16–15:25:47), all with `strategy="unknown"` and `external_close`. Bot placed a SPY exit order, then ghost cleanup removed the local position before the fill was detected. `_register_opening_fill` re-created it as a phantom long → override fired again.

**Root cause:** `_fast_step_position_sync` ghost cleanup ran in the same fast cycle as the exit order fill, removing a position the bot had just sold. Alpaca paper trading fills market orders nearly instantly, so the position disappears from broker before the next fill detection cycle.

**Culprit:**
- `orchestrator.py:959` — Ghost cleanup did not check whether a pending exit order existed for the symbol before removing the local position.

**Fix:** Ghost cleanup now defers removal for any symbol where `can_place_order(symbol) == False` (i.e., a pending or partially-filled exit order is in flight).

## Bug 6 — META phantom journal entries (entry_time == exit_time)

**Severity:** Medium
**Evidence:** 5 META journal entries with `entry_time == exit_time` (within 373µs–1ms), all `entry_price=628.23`, `exit_price=620.0`, `pnl=1.31%`, `signals_at_entry={}`, `claude_conviction=0.0`, `strategy="swing"`, `stop=635`, `target=615`. No META orders in `orders.json`. The last phantom was at `23:37:30 UTC` (19:37 ET — post-market hours, well after the bot's `is_market_open()` gate would have shut down all loop activity).

**Root cause:** `TradeJournal` hardcoded its file path to `ozymandias/state/trade_journal.jsonl` regardless of how the Orchestrator's state manager was configured. The test fixture redirected `_state_manager._dir = tmp_path` so all portfolio/order reads and writes went to a temp directory, but `_trade_journal._path` was never redirected and always pointed to the real state file.

`TestDispatchConfirmedFill` (test_orchestrator.py line 1088) creates a META short position with `avg_cost=628.23`, `stop_loss=635`, `profit_target=615` in the test portfolio, then calls `_dispatch_confirmed_fill` with `fill_price=620.0`. This routes to `_journal_closed_trade` (line 765), which appended to the real `trade_journal.jsonl` — identical values to every phantom.

The 15:22–15:39 phantoms were written during pytest runs while the trading session was active. The 23:37 UTC phantom was written by a pytest run ~3.5 hours after market close.

**Culprit:**
- `trade_journal.py:19–20` — Module-level `STATE_DIR` and `TRADE_JOURNAL_FILE` constants; `TradeJournal.__init__` used `TRADE_JOURNAL_FILE` when no path was provided.
- `orchestrator.py:127` (pre-fix) — `TradeJournal()` created without a path, with no link to the state manager's directory.
- `tests/test_orchestrator.py:77` and `tests/test_integration.py:154` — Both fixtures redirected `_state_manager._dir` but not `_trade_journal._path`.

**Fix:**
1. `orchestrator.py` — `TradeJournal` now constructed with `path=self._state_manager._dir / "trade_journal.jsonl"`, making the path relationship to the state manager explicit.
2. Both test fixtures now redirect `o._trade_journal._path = tmp_path / "trade_journal.jsonl"` alongside `_state_manager._dir`, ensuring test journal writes are fully isolated.

## Bug 7 — Portfolio cash and buying_power never synced from broker

**Severity:** Medium
**Evidence:** `portfolio.json` shows `cash: 0.0`, `buying_power: 0.0` for the entire session. Claude's context prompt correctly received live broker figures ($59,603 buying power) but the persisted state was always zero.

**Root cause:** No code path updated `portfolio.cash` or `portfolio.buying_power` from the broker account. The medium loop fetched `acct = await self._broker.get_account()` for use in ranking and Claude context, but never wrote the values back to portfolio state.

**Culprit:**
- `orchestrator.py:1271` (pre-fix) — Medium loop step 4 used `acct` for ranking but did not update portfolio.

**Fix:** After fetching `acct`, the medium loop now updates and saves portfolio cash and buying_power if they differ from broker-reported values (line 1273).

## Bug 8 — Exit price recorded as $0 for ghost-cleaned positions

**Severity:** Low
**Evidence:** NVDA journal entry shows `exit_price: 0.0`, `pnl_pct: 0.0` despite being a 32-share position entered at $184.81. Entry was via position adoption (strategy="unknown", no signals).

**Root cause:** Ghost cleanup uses `_latest_indicators[symbol].get("price", 0.0)` for the exit price. NVDA's indicators had not been populated yet (medium loop hadn't run a full scan for NVDA before the position was ghost-cleaned), so the default 0.0 was used.

**Culprit:**
- `orchestrator.py:1010` (pre-fix) — No fallback for missing price in ghost cleanup journal write.

**Fix:** Falls back to `pos.avg_cost` when `current_price == 0.0`, recording pnl=0% rather than a meaningless $0 exit (line 1014).

## Bug 9 — PDT guard undercounts day trades relative to broker

**Severity:** Medium
**Evidence:** Startup log at 16:05 showed `daytrades_used=5` (broker). Bot's `count_day_trades()` returned `1` (NVDA round-trip only). Divergence of 4: AMD's 16 short-sell orders and SPY activity executed at broker level never produced matching buy+sell pairs in `orders.json`.

**Root cause:** `PDTGuard.count_day_trades()` counts only from `orders.json` (orders the bot placed). Phantom/adoption trades that hit the broker but weren't correctly journaled (e.g. AMD 16 sells with no buy, SPY buys from ghost cleanup race) are invisible to this counter. On restart, the bot would have believed it had 2 day trades remaining when it actually had none.

**Culprit:**
- `pdt_guard.py:124` — `count_day_trades` only counts local orders; no reference to broker-reported `acct.daytrade_count`.
- `orchestrator.py:253` — `PDTGuard` created but never seeded with the startup broker account's `daytrade_count`.

**Fix:** Added `PDTGuard.broker_floor: int = 0`. `count_day_trades` now returns `max(local_count, broker_floor)`. Orchestrator seeds `broker_floor = acct.daytrade_count` at startup (line 255) and refreshes it every medium loop cycle after the account fetch.

## Anomaly 10 — VIX added to tier-1 watchlist as a non-tradeable index

**Severity:** Medium
**Evidence:** `watchlist.json` entry for `VIX` with `priority_tier: 1`, timestamp `2026-03-16T16:42:36`. 12+ repeated WARNING entries at ~2-minute intervals in `logs/current.log` (16:44–16:59 ET): `yfinance ERROR: $VIX: possibly delisted; no price data found`. No order was placed, but the placement risk existed as long as VIX remained in the watchlist.

**Root cause:** The watchlist prompt had no rule against non-tradeable index tickers. Claude correctly identifies VIX as a volatility measure but does not know that it cannot be ordered on Alpaca.

**Culprits:**
- `config/prompts/v3.3.0/watchlist.txt` — No instruction prohibiting index tickers.
- `orchestrator.py:_apply_watchlist_changes` — No code-level guard against index symbols before appending to watchlist.

**Fix:**
1. Added an explicit rule to `watchlist.txt`: do not add VIX, VXN, SPX, NDX, RUT, DJI, COMP or any `^`-prefixed symbol; use tradeable proxies (e.g. UVXY/VIXY for volatility).
2. Added `_INDEX_BLACKLIST` set in `_apply_watchlist_changes`; any symbol in the set or starting with `^` is logged as a warning and skipped before being appended.

## Anomaly 11 — 17 ETFs occupying tier-1 in persisted watchlist state

**Severity:** Medium (state artifact; no code fix required)
**Evidence:** `watchlist.json` shows all 45 symbols at `priority_tier: 1`, including SPY, QQQ, IWM, DIA, VTI, XLF, XLK, XLE, XLV, XLI, XLU, XBI, SMH, GLD, GDX, USO, and VIX — all of which should be tier-2 or blacklisted.

**Root cause:** The watchlist state was last written before the prompt fix (commit `88548f0`) that enforced ETFs → tier-2. The prompt fix only affects newly-generated watchlists; the persisted state retains the old tier assignments until the slow loop's next watchlist rebuild triggers a full refresh.

**Fix:** None needed. The state will self-correct on the next `run_watchlist_build()` call (triggered when the watchlist shrinks below 10 symbols or on a forced rebuild). The prompt and code fixes from Anomaly 10 will apply at that point.

## Anomaly 12 — `last_evaluated` field never populated after creation

**Severity:** Low
**Evidence:** All 45 `watchlist.json` entries have `last_evaluated: ""`. Grep of entire codebase shows the field was defined in `WatchlistEntry` and deserialized in `_from_dict_watchlist_entry`, but no code path ever wrote to it after initial construction.

**Fix:** Removed `last_evaluated` from `WatchlistEntry` dataclass and from `_from_dict_watchlist_entry`. Existing JSON entries that carry the field will silently ignore it on load (the `.get()` key is simply absent from the constructor).

## Anomaly 13 — `signals_at_entry` empty on 12 of 13 journal entries

**Severity:** Low (expected; no fix required)
**Evidence:** Only journal entry #10 (SPY short, 15:25:47) has non-empty `signals_at_entry`, non-zero `claude_conviction` (0.55), and non-zero `composite_score` (0.516). All other entries have empty dicts and zero scores.

**Root cause:** `_journal_closed_trade` reads signal context from `_entry_contexts[symbol]`, which is populated only by `_register_opening_fill`. All 12 affected trades were adopted or phantom positions that bypassed the normal opening fill path, so `_entry_contexts` was never populated for them.

**Fix:** None — expected behavior for adopted positions. Future sessions will show richer signal data once the adoption bugs (1–5) are resolved and all positions enter through the normal pipeline.

### Closed: AMD position still open at session end

As of 16:47 ET, the 2-share AMD short (adopted at 15:10:42, no stop/target) was still open. Resolved by the combination of Bug 2 + Bug 3 fixes:

- **Bug 3** ensures adoption sets `direction="short"` correctly, so `_fast_step_quant_overrides` skips it (short positions are intentionally excluded from VWAP/ATR override logic at line 866).
- **Bug 2** ensures that when the slow-loop Claude review fires and returns `action="exit"`, a market buy-to-cover order is placed immediately.

Short positions with no stop/target have no quant override exit path by design — they rely entirely on Claude's slow-loop review. The slow loop has a time-ceiling trigger that guarantees Claude is called within a bounded interval regardless of other triggers, so the position will be caught and exited on the next bot restart.
