# Bug Report — 2026-03-24

Professional audit of codebase and prompts. 12 verified findings after false-positive review.

---

## CRITICAL

### BUG-001: `bypass_market_hours: true` active in production config
**File:** `ozymandias/config/config.json:24`
**Severity:** CRITICAL

`"bypass_market_hours": true` is set in the production `config.json`. This flag disables all
market-hours guards in the orchestrator, allowing entries and exits to be attempted outside of
regular trading hours (premarket, afterhours, weekends). Alpaca will reject market orders outside
regular hours, but limit orders may be routed to extended-hours sessions with significantly worse
fills, wider spreads, and no liquidity. More critically, the session filter for momentum entries
(`blocks_eod_entries`) is also bypassed, so EOD close protection no longer applies.

This flag is likely a development artifact left enabled after debugging. It must be `false` in any
session where real capital (even paper capital) is at risk.

## HIGH

### BUG-002: Gemini fallback holds `_call_lock` for up to 120 seconds
**File:** `ozymandias/intelligence/claude_reasoning.py`
**Severity:** HIGH

The `_call_lock` asyncio serialization added in this session wraps the entire retry loop in
`call_claude`. When Claude is overloaded and falls through to the Gemini fallback path,
`_call_gemini_fallback` can run for up to 120 seconds (its own internal timeout). During this
entire period the lock is held, blocking all other Claude calls — including fast-loop emergency
checks, watchlist builds, and position reviews that may be queued.

The lock was designed to enforce a 3-second inter-call gap, not to serialize an entire fallback
session. The Gemini call should either (a) acquire a separate lock, (b) release the primary lock
before entering the fallback, or (c) the `finally:` block updating `_last_call_end_time` should
run before the fallback is attempted so other callers can proceed while Gemini is in-flight.

### BUG-003: Token budget overflow is silent when positions alone exceed the budget
**File:** `ozymandias/intelligence/claude_reasoning.py:498–503`
**Severity:** HIGH

The token budget calculation subtracts the prompt template size from `_TOTAL_TOKEN_BUDGET` to
determine how many watchlist symbols to include. However, if the position context (8 positions ×
~800 tokens each = ~6,400 tokens) plus the template size already exceeds the budget, the
computed symbol quota becomes negative. The code clamps this to zero, so no watchlist symbols are
included, but no warning is logged and Claude is called anyway — with a context that contains only
positions and no candidates to evaluate. Claude will produce an empty `new_opportunities` list
every cycle without any diagnostic signal that the budget is exhausted.

### BUG-004: `rsi_slope_max` accepts positive values and silently blocks all short entries
**File:** `ozymandias/intelligence/opportunity_ranker.py:164–173`
**Severity:** HIGH

The `evaluate_entry_conditions` function checks the `rsi_slope_max` gate for short entries:

```python
if "rsi_slope_max" in conditions:
    threshold = conditions["rsi_slope_max"]
    if signals.get("rsi_slope_5", 0) > threshold:
        return False, f"rsi_slope_max gate: slope {slope:.2f} > {threshold}"
```

The prompt instructs Claude to supply a *negative* value (e.g. `-0.5` means "RSI must be falling
by at least 0.5 per bar"). However, there is no validation that the value is negative. If Claude
emits a positive value (e.g. `0.5` due to hallucination or prompt ambiguity), the gate becomes
`slope > 0.5` — which passes only when RSI is rising rapidly, the exact opposite of a short setup,
and silently blocks every valid short entry that cycle. The bug is detectable only by checking
`recommendation_outcomes` for repeated `conditions_waiting` on short candidates.

## MEDIUM

### BUG-005: Prompt PLTR mean-reversion example — `require_macd_histogram_expanding: false` is a no-op
**File:** `ozymandias/config/prompts/v3.6.0/reasoning.txt`
**Severity:** MEDIUM

The mean-reversion fade example in the prompt reads:

```
Mean-reversion fade example (PLTR extended): require_above_vwap: true, rsi_min: 68,
rsi_slope_max: -0.3, require_macd_histogram_expanding: false
```

`require_macd_histogram_expanding: false` does not assert that the histogram is *contracting* — it
simply skips the expanding check entirely, the same as omitting the key. To express "histogram must
be contracting or flat", the prompt would need to document a `require_macd_histogram_contracting`
key (which does not exist) or note that `false` is not a meaningful constraint. As written, a
reader of the example may believe `false` acts as an inversion gate, leading to misspecified
`entry_conditions` for fade setups.

### BUG-006: Momentum strategy docstring RSI range contradicts live config
**File:** `ozymandias/strategies/momentum_strategy.py` (module docstring / inline comments)
**Severity:** MEDIUM

The strategy docstring and inline comments reference an RSI entry range of approximately 45–65.
The live `strategy_params.momentum` config has `rsi_entry_min: 60`, tightened after the March 19
backtest (Phase 16 post-MVP hardening). The docstring was not updated. This creates a discrepancy
between documentation and actual behavior that may mislead prompt tuning or future code changes
that rely on the documented range.

### BUG-007: Relationship between `min_volume_ratio` and `min_rvol_for_entry` is undocumented
**File:** `ozymandias/strategies/momentum_strategy.py`, `ozymandias/intelligence/opportunity_ranker.py`
**Severity:** MEDIUM

`min_rvol_for_entry` is the strategy-level gate applied in `apply_entry_gate`. `min_volume_ratio`
(or `require_volume_ratio_min` in `entry_conditions`) is the Claude-specified per-trade gate
evaluated in `evaluate_entry_conditions`. Both gate on RVOL but are evaluated at different points
with no documented precedence or interaction. A position can pass the ranker's entry-conditions
check but fail the strategy gate, or vice versa, with no log distinguishing which gate rejected.
An operator tuning RVOL thresholds may adjust one without knowing the other exists.

### BUG-008: Swing strategy has no direction-aware entry gate
**File:** `ozymandias/strategies/swing_strategy.py`
**Severity:** MEDIUM

`SwingStrategy.apply_entry_gate` only blocks longs in bearish-aligned trend and shorts in
bullish-aligned trend (the `block_bearish_trend` param). It has no VWAP relationship check, no RSI
slope check, and no RVOL floor — unlike `MomentumStrategy` which has all three. For short swing
entries specifically, there is no strategy-level protection against entering a short when price is
extended above VWAP in a bullish-aligned trend (a potential mean-reversion setup that the swing
strategy should not be trading). This gap means Claude's `entry_conditions` is the only gate for
swing shorts, with no strategy-layer backstop.

### BUG-009: `rsi_slope_5` spans overnight gaps, contaminating first-bar-of-day slope values
**File:** `ozymandias/intelligence/technical_analysis.py` (RSI slope calculation)
**Severity:** MEDIUM

`rsi_slope_5` is computed as a linear regression or finite difference over the last 5 RSI values
in the intraday bar series. At market open, the first 5 bars of the day are joined to the last bars
of the previous session's cached data. If the overnight gap caused a large RSI jump (e.g. strong
premarket move), the computed slope for the first several bars of the new session will be
artificially steep — either strongly positive (gap up) or strongly negative (gap down) — and will
not reflect current intraday momentum. Gates like `rsi_slope_min: 0.5` or `rsi_slope_max: -0.5`
may fire or block erroneously in the first 5–10 minutes of each session.

### BUG-010: `expected_direction` string is not validated on watchlist state load
**File:** `ozymandias/core/orchestrator.py` or `ozymandias/execution/state_manager.py` (watchlist load path)
**Severity:** MEDIUM

`WatchlistEntry.expected_direction` is added by Claude via `watchlist_changes.add` responses.
The value is written to `watchlist.json` and reloaded on startup. There is no schema validation
that the loaded string is one of `{"long", "short", "either"}`. An invalid value (typo, model
hallucination, manual edit) will propagate silently through `compute_composite_score` direction
adjustment logic and `ta_readiness` assembly, potentially producing incorrect composite scores
for affected symbols across the entire session without any error log.

### BUG-011: `_recently_closed` fill-protection guard is vulnerable to clock skew on restart
**File:** `ozymandias/core/orchestrator.py` (`_recently_closed` persistence path)
**Severity:** MEDIUM

`_recently_closed` is persisted to state and restored on restart. The expiry window is computed
as `close_time + _recently_closed_ttl_sec` against `time.time()` (wall clock). If the host clock
is stepped backward (NTP correction, DST edge case, container restart with stale clock), entries
that should have expired may be treated as still active, blocking re-entry into recently closed
symbols for longer than intended. Conversely, a forward clock jump may expire entries prematurely,
removing fill protection before the TTL has actually elapsed. The window is short (~60s default)
so impact is limited, but a large NTP correction at startup could affect the entire first session.

## LOW

### BUG-012: Tier-1 composite score ranking is biased toward longs for `expected_direction: "either"` symbols
**File:** `ozymandias/intelligence/opportunity_ranker.py` (composite score direction adjustment)
**Severity:** LOW

When `expected_direction` is `"either"`, `compute_composite_score` defaults to evaluating signals
in the long direction (bullish MACD, above-VWAP position, positive RSI slope all score positively).
Bearish signals on an `"either"` symbol will therefore score lower than bullish signals of equal
magnitude, making bearish `"either"` candidates less likely to surface in tier-1 context for Claude
than equivalent directional short candidates. In a broadly bearish session where `"either"` symbols
are actually short candidates, their composite scores are suppressed and Claude may not see them.
The correct behavior would be to score `"either"` symbols against both directions and take the
higher composite score, or to let Claude re-direct them via `expected_direction` update before
the score is used for ranking.
